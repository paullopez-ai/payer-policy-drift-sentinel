# CLAUDE.md - payer-policy-drift-sentinel

## Project Identity

Payer Policy Drift Sentinel is a healthcare payer AI prototype that detects policy, denial, authorization, appeal, and provider friction drift using real Snowflake, Azure Databricks, and Azure infrastructure for the live demo path. Snowflake is the governed synthetic payer data plane. Azure Databricks is the analytics and AI engineering plane. Azure OpenAI generates supervised, cited evidence packets through a LangGraph workflow that includes quality gates, cost checks, audit logging, and human review.

## Stack

- Runtime: Python 3.12 with uv
- API: FastAPI
- Workflow: LangGraph StateGraph
- Contracts: Pydantic v2
- Data platform: Snowflake, live track only
- Analytics platform: Azure Databricks, live track only
- LLM SDK: Azure OpenAI, live track only
- Infrastructure: Terraform
- Testing: pytest
- UI companion: Next.js 16+, separate repo

## Critical Constraints

- Python version must be pinned to `>=3.12,<3.13` in `pyproject.toml`. Do not use 3.13 or 3.14. Run `uv python pin 3.12` if uv selects a newer version.
- No proprietary data, no PHI, no real patient data, no real payer policy text.
- All demo data must be synthetic or public-safe.
- Test mode must never call Snowflake, Databricks, Azure OpenAI, or Azure APIs.
- `MOCK_LLM=true` and `EXTERNAL_CLIENT_MODE=mock` are required for tests.
- Live demo mode may call Snowflake, Azure Databricks, Azure OpenAI, and Azure-hosted services.
- Secrets must live in environment variables, local ignored files, Azure Key Vault, or Terraform variables that are not committed.
- Never commit `.env`, `*.tfvars`, Terraform state, private keys, Snowflake keys, or Databricks tokens.
- All model outputs must conform to Pydantic contracts.
- Human review is mandatory for restricted trust boundary findings.
- Terraform deployment and destruction are manual execution gates.

## Architecture Principles

This project separates detection from explanation. Snowflake holds governed synthetic payer facts, Databricks performs feature engineering and drift scoring, and Azure OpenAI explains findings only after structured evidence is available. The workflow is decomposed into explicit LangGraph nodes so each stage can be tested, audited, costed, and blocked independently.

## Test Track vs. Live Demo Track

Test Track:
- `MOCK_LLM=true`
- `EXTERNAL_CLIENT_MODE=mock`
- No cloud calls
- Mock Snowflake, Databricks, and Azure OpenAI clients
- Used for pytest and CI only

Live Demo Track:
- `MOCK_LLM=false`
- `EXTERNAL_CLIENT_MODE=live`
- Uses real Snowflake tables, real Azure Databricks jobs, real Azure OpenAI deployment, and real Azure Container Apps deployment
- Requires Terraform apply and destroy

## Autonomous Build Rule

Work through the build sequence phase by phase without stopping to ask for confirmation between steps, unless you hit a Manual Execution Gate. Complete each phase fully before starting the next. When a phase produces a runnable artifact, confirm it runs before moving on. Run dependency installation, the test suite in mock mode only, and API smoke tests autonomously. Never attempt to run dataset generation for live Snowflake, Databricks jobs, evaluation scripts against live services, Terraform apply, or Terraform destroy autonomously.

## Manual Execution Gates

Gate 1 - after Terraform files are written:
  Command:
    cd infra/terraform && terraform init && terraform fmt -recursive && terraform validate && terraform plan -out tfplan
  Expected output:
    Terraform initializes, formats, validates, and creates a plan without applying resources.
  Confirm and I will continue to live deployment documentation.

Gate 2 - live infrastructure deployment:
  Command:
    cd infra/terraform && terraform apply tfplan
  Expected output:
    Azure, Databricks, and Snowflake demo resources are created. Terraform outputs include API URL placeholders, Databricks workspace URL, Key Vault name, and Snowflake database name.
  Confirm and I will continue to Snowflake seeding.

Gate 3 - seed Snowflake:
  Command:
    uv run python scripts/seed_snowflake.py --scenario-set full --seed 42 --warehouse PAYER_DRIFT_WH --database PAYER_DRIFT_SENTINEL
  Expected output:
    Synthetic Snowflake tables are populated and row counts are printed.
  Confirm and I will continue to Databricks job execution.

Gate 4 - run Databricks jobs:
  Command:
    uv run python scripts/run_databricks_jobs.py --job drift_feature_pipeline --wait
  Expected output:
    Databricks drift features and drift findings are produced.
  Confirm and I will continue to live API smoke testing.

Gate 5 - live API smoke test:
  Command:
    uv run python scripts/live_smoke_test.py --api-url "$DRIFT_SENTINEL_API_URL" --scenario-id denial-spike-imaging
  Expected output:
    API returns an evidence packet with citations, confidence, cost, and review status.
  Confirm and I will continue to README finalization.

Gate 6 - teardown:
  Command:
    cd infra/terraform && terraform destroy
  Expected output:
    Azure, Databricks, and Snowflake demo resources are destroyed or removed according to Terraform configuration.
  Confirm and I will mark live demo teardown complete.
